1. System Architecture

CCRTS follows a 3-Tier Architecture — Presentation, Application, and Data layers are fully separated, allowing independent scaling and maintenance.

Presentation Layer (Frontend)

React.js SPA running on Vite dev server (port 5173). Communicates with backend exclusively via Axios HTTP calls.

React 19 Vite 8 React Router DOM v7 Axios React Icons Pure CSS Charts
Application Layer (Backend)

Node.js + Express v5 REST API server running on port 3001. Handles authentication, business logic, file uploads, notifications, and background SLA jobs.

Node.js 24 Express v5 JWT (jsonwebtoken) bcryptjs Multer v2 dotenv CORS
Data Layer (Database)

SQLite database accessed via Node.js built-in DatabaseSync API. Single file DB (ccrts.db) — no external DB server required.

SQLite (node:sqlite) DatabaseSync API ALTER TABLE Migrations CHECK Constraints

3. Module Breakdown

Auth Module

  • User registration & login
  • JWT token generation
  • protect middleware
  • requireRole middleware

Complaint Module

  • Create / view / update complaints
  • Auto-generate CMP-XXX IDs
  • Status transitions with history
  • SLA deadline computation

SLA & Escalation Module

  • Background SLA checker (5 min)
  • Auto-escalation on breach
  • Escalation dashboard view
  • Supervisor reassignment

Notification Module

  • DB-stored notifications
  • 30s frontend polling
  • Mark read / mark all read
  • Complaint deep-link on click

Attachment Module

  • Multer file upload (5MB max)
  • Allowed: PDF, JPG, PNG, DOC, TXT
  • Static file serving from /uploads
  • Download links in complaint detail

Feedback Module

  • Star rating (1–5) with comments
  • Only for Resolved/Closed complaints
  • Duplicate submission prevention
  • Aggregated in Reports

Reports Module

  • Single aggregated API endpoint
  • Overview, by category, by priority
  • Monthly trend (6 months)
  • Agent performance + satisfaction
  • Cross-category assignment log

User Management Module

  • Admin-only user list
  • Role change per user
  • User deletion (self-protected)
  • Role badge display

Agent Category Module

  • Agent-to-category mapping management
  • Last-agent removal guard per category
  • Category-filtered assignment dropdown
  • Cross-category override with audit log

6. Complaint Lifecycle Design

The complaint status is a controlled state machine. Invalid transitions are blocked at the API layer. The history table records every transition.

From StatusAllowed Next StatusTriggered ByConditions
OpenAssignedAdmin / SupervisorAuto-set when agent is assigned
AssignedIn ProgressSystemAuto-set when assigned agent adds first note
In ProgressPending Customer Response, Resolved, EscalatedAssigned Agent / SystemResolved: assigned agent only; note or attachment required. Pending: mandatory note
Pending Customer ResponseIn Progress, Resolved, EscalatedAssigned Agent / SystemResolved: assigned agent only; note or attachment required
EscalatedIn Progress, ResolvedAdmin / Supervisor / Assigned AgentResolved: only by the assigned agent with evidence (note or attachment)
ResolvedClosedAdmin / Supervisor / AgentClosing rules depend on feedback rating (see RS §6.5)
ClosedAssigned (reopen)Admin / Supervisor onlyMandatory reason note required; resolved_at cleared; agent notified; other management role notified
Closed— (terminal state)——

Phase 2 — ETL Pipeline & Analytics

Phase 2 extends the platform with an ETL pipeline that imports synthetic complaint data, populates pre-computed reporting tables, and surfaces analytics through the existing Reports page.

ETL Pipeline Architecture

Step 1 — Extract

etl/generate_dataset.js generates complaints_dataset.csv (250 records) and feedback_dataset.csv (~106 records) with statistically distributed priorities, statuses, SLA deadlines, and agent workloads. Active complaints use timestamps anchored to "now" so SLA deadlines are always in the future at generation time.

Step 2 — Transform

etl/etl_import.js validates each row (required fields, enum membership, FK resolution for customer/agent/category), maps email addresses to user IDs, and skips duplicates. Invalid rows are logged and excluded — the pipeline is non-fatal and idempotent.

Step 3 — Load & Refresh

etl/etl_refresh_reports.js truncates and repopulates all 4 reporting tables in a single transaction. Safe to re-run at any time. etl/reset.js is a full demo-reset command — deletes all CMP-016+ records and reruns the full pipeline.

Demo usage: Run node etl/reset.js before any demo. Active complaints are regenerated with fresh timestamps. The data has a natural 72-hour shelf life before Low-priority complaints start auto-escalating.

Reporting Tables

Four pre-computed analytics tables are populated by the ETL refresh. The Reports API reads from these tables instead of running live aggregation queries on the complaints table.

TableGranularityKey Metrics
report_sla_summaryperiod × category × prioritytotal, compliant, breached, compliance_rate
report_category_trendsperiod × categorytotal, resolved, escalated, avg_resolution_hours
report_agent_performanceperiod × agenttotal_handled, avg_rating, sla_compliance, reopened, score (0–100)
report_resolution_trendsperiod × prioritytotal_resolved, avg/min/max resolution hours

Agent Performance Score

A composite 0–100 score is computed per agent per month and stored in report_agent_performance.score. It is also computed in real time for the agent's own sidebar badge (last 30 days).

ComponentWeightFormula
Customer Rating50 pts(avgRating / 5) × 50
SLA Compliance30 pts(slaCompliance / 100) × 30
Reopen Rate20 pts(1 − reopenRate) × 20

Score thresholds: ≥ 80 green (good) · ≥ 60 orange (watch) · < 60 red (at risk). The score badge appears in the sidebar for agent-role users and in the Agent Performance tables on Admin and Quality dashboards.
