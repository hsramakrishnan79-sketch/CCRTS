1. Introduction

1.1 Project Title

Customer Complaint & Resolution Tracking System (CCRTS)

1.2 Project Description

The Customer Complaint & Resolution Tracking System is a centralized web-based application designed to help organizations efficiently manage, monitor, and resolve customer complaints and service issues.

The platform enables customers, support teams, service agents, supervisors, and administrators to log complaints, assign issues, track progress, communicate updates, and ensure timely resolution of customer concerns.

1.3 Applicable Industries

  • Telecom & IT Support Services
  • Banking & Financial Services
  • Retail & E-Commerce
  • Healthcare & Education
  • Logistics & Utility Services

2. Business Need

Organizations often struggle with customer complaint management because complaints are tracked manually using emails, spreadsheets, or disconnected systems. This creates several operational challenges:

  • Delayed issue resolution
  • Poor complaint visibility across teams
  • Missed SLA commitments
  • Lack of accountability
  • Duplicate complaint handling
  • Inefficient communication with customers
  • Difficulty tracking complaint history
  • Low customer satisfaction
The CCRTS addresses these challenges by providing a unified platform for complaint registration, workflow management, escalation handling, resolution tracking, and reporting.

6. Functional Requirements

6.1 User Authentication & Authorization

  • User registration with role assignment
  • Secure login and logout with JWT session management
  • Role-based access control (RBAC) for all routes
  • Password encryption using bcrypt
  • Automatic session expiry and redirect on token expiry

6.2 Complaint Registration

  • Register new complaint with category and description
  • Auto-generate unique complaint ID (format: CMP-XXX)
  • Select complaint category from predefined list
  • Upload supporting attachments (PDF, JPG, PNG, DOC, TXT — max 5MB)
  • Priority is not set at submission — admin/supervisor assigns it after reviewing the complaint
  • SLA deadline is computed and stored when admin/supervisor sets the priority

6.3 Complaint Workflow Management

  • Assign complaints to specific agents
  • Update complaint status through defined transitions
  • Add notes/comments at each status change
  • Full audit trail of all status changes with timestamps
  • View complete complaint history timeline

6.4 SLA & Escalation Management

  • SLA deadline computed and stored when admin/supervisor sets complaint priority
  • Automatic escalation when SLA deadline is breached (background job)
  • SLA background job runs every 5 minutes
  • Escalation notifications sent to supervisor and admin
  • Escalation dashboard for supervisors to manage and reassign

6.5 Complaint Resolution

  • Only the assigned agent can mark a complaint as Resolved — admin and supervisor are blocked; other agents are also blocked
  • A note or uploaded attachment is required as resolution evidence before the status is accepted
  • Customer receives notification on resolution
  • Customer submits star rating feedback (1–5) after resolution
  • Closing rules enforced based on feedback rating:
    • Rating 3–5 (satisfied): agent, admin, or supervisor can close
    • Rating 1–2 (dissatisfied): admin/supervisor only, mandatory note required
    • No feedback, < 7 days since resolved: closing blocked for all roles
    • No feedback, ≥ 7 days since resolved: admin/supervisor force-close with mandatory note
  • Complaint reopen (Closed → Assigned):
    • Only admin or supervisor can reopen a closed complaint
    • Mandatory reason note required — empty note is blocked
    • Status returns to Assigned so admin/supervisor can reassign if needed before the agent resumes work
    • resolved_at is cleared on reopen
    • Assigned agent is notified; the other management role (admin notifies supervisor, supervisor notifies admin) is always informed

6.6 Notification Management

  • In-app notifications with 30-second polling
  • Unread count badge on notification bell
  • Mark individual or all notifications as read
  • Notification triggers: complaint creation, assignment, status update, escalation, resolution

6.7 Dashboard & Analytics

  • Role-specific dashboards (admin/supervisor/agent/customer views)
  • Complaint statistics: total, open, resolved, escalated, reopened count
  • SLA breach count and on-time resolution rate
  • Average resolution time
  • Agent performance table
  • Monthly complaint trend chart (last 6 months)
  • Complaints by category and priority charts
  • Customer satisfaction rating distribution
  • Cross-category assignment count in overview; breakdown by overloaded category and by agent; full override log with reason and timestamp

6.8 Feedback Management

  • Customer submits feedback only for Resolved or Closed complaints
  • Star rating from 1 to 5
  • Optional text comments
  • One feedback per complaint (duplicate prevention)
  • Aggregate satisfaction metrics shown in Reports

6.9 User Management

  • Admin can view all registered users
  • Admin can change any user's role
  • Admin can delete users (self-deletion blocked)
  • Admin cannot change their own role

6.10 Agent-Category Management

  • Each complaint category must have at least one mapped agent at all times
  • Admin manages agent-to-category mappings via a dedicated management page
  • Admin can add an agent to a category or remove an agent from a category
  • Removing the last agent from a category is blocked — the system returns an error
  • When assigning a complaint, only agents mapped to the complaint's category appear in the dropdown
  • Cross-category override (admin or supervisor only):
    • Either role can assign an agent outside their mapped category via an explicit override action
    • Mandatory reason note required — empty note is blocked at both application and database level
    • The other management role is notified with the agent name and reason
    • Assigned agent receives a notification identifying the assignment as cross-category
    • Every override is permanently logged in cross_category_assignments with full audit detail

11. Agent-Category Management

11.1 Category Coverage Rule

Every one of the 7 complaint categories must have at least one agent mapped to it at all times. A category with no agents cannot accept complaint assignments, which is an unacceptable service gap. The system enforces this rule at the API layer — the last agent in a category cannot be removed.

11.2 Assignment Enforcement

When admin or supervisor assigns an agent to a complaint, only agents mapped to the complaint's category are shown in the assignment dropdown. The backend also enforces this — assigning an unmapped agent without an explicit override flag returns a 400 error.

11.3 Cross-Category Override

Cross-category overrides are intended for exceptional situations — category agent overload, specialist unavailability, or urgent escalations. They are not routine assignments. The mandatory note, peer notification, and permanent log create the accountability needed for responsible use.
RuleDetail
Who can overrideAdmin or supervisor (either one is sufficient — joint approval is too slow for escalation scenarios)
Mandatory noteRequired — empty note is rejected. Note must explain why the override is needed.
Peer notificationAdmin overrides → supervisor notified. Supervisor overrides → admin notified. Ensures dual visibility without requiring dual approval.
Agent notificationAgent receives a distinct message identifying the assignment as cross-category so they know the complaint type may be outside their usual scope.
Permanent logEvery override written to cross_category_assignments with full detail: complaint, agent, bypassed category, assigner, reason, timestamp.
ReportingCross-assignment count shown in Reports overview; breakdown by overloaded category and by agent; full log visible to admin, supervisor, and quality team.
