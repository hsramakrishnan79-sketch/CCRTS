1. Database Overview

CCRTS uses SQLite as its database engine, accessed via the Node.js built-in node:sqlite module using the synchronous DatabaseSync API. The database is a single file (backend/database/ccrts.db) that requires no external DB server to run.

The database is fully initialized on server startup via backend/config/db.js using CREATE TABLE IF NOT EXISTS statements, making deployment straightforward — no migration runner needed for fresh installs.

11. Table: agent_categories

Junction table implementing the many-to-many relationship between agents and complaint categories. An agent can be mapped to multiple categories; a category can have multiple agents. The business rule that every category must have at least one agent is enforced at the application layer — the API blocks removal of the last agent from any category.

agent_categories

2 columns — composite PK
ColumnTypeConstraintsDescription
agent_idINTEGERNOT NULL FK → users.id PK (composite)The agent being mapped
category_idINTEGERNOT NULL FK → categories.id PK (composite)The category the agent is mapped to
CREATE TABLE IF NOT EXISTS agent_categories (
  agent_id     INTEGER NOT NULL REFERENCES users(id),
  category_id INTEGER NOT NULL REFERENCES categories(id),
  PRIMARY KEY (agent_id, category_id)
);

12. Table: cross_category_assignments

Permanent audit log for every cross-category assignment override. Written only when admin or supervisor assigns an agent to a complaint outside the agent's mapped categories. The note column is NOT NULL at the database level — an override without a reason cannot be inserted. The separate table avoids a sparse boolean column in complaint_history and enables direct cross-assignment reporting without text parsing.

13. SLA Rules Reference

SLA deadlines are not stored in a separate table — they are computed when admin/supervisor sets the priority and stored in complaints.sla_deadline. The column is NULL until priority is assigned. This approach prevents the SLA clock from starting before someone has actually reviewed the complaint.

PrioritySLA HoursFormulaExample
Critical4 hourscreated_at + 4hFiled at 10:00 AM → deadline 2:00 PM
High24 hourscreated_at + 24hFiled Monday 10 AM → deadline Tuesday 10 AM
Medium48 hourscreated_at + 48hFiled Monday → deadline Wednesday
Low72 hourscreated_at + 72hFiled Monday → deadline Thursday
The SLA background job (runs every 5 minutes) queries:

SELECT * FROM complaints WHERE sla_deadline < datetime('now') AND status NOT IN ('Resolved','Closed','Escalated')

Any matching complaints are automatically updated to 'Escalated' and notifications are sent to all supervisors and admins.

Phase 2 — Reporting Tables

Four pre-computed analytics tables were added in Phase 2. They are populated by the ETL refresh script (etl/etl_refresh_reports.js) which truncates and repopulates all four tables in a single transaction. These tables are read-only from the application layer — all writes go through the ETL pipeline.

These tables use UNIQUE constraints on their natural keys so that INSERT OR REPLACE can safely re-run without duplicates. The refreshed_at column on every row records when the ETL last ran.

report_sla_summary

Monthly SLA compliance aggregated by category and priority. Covers resolved/closed complaints only — open complaints are excluded since SLA is only measurable after resolution.

14. Migration Strategy

All tables use CREATE TABLE IF NOT EXISTS — safe to run on both fresh and existing databases. The database file is created automatically on first server start; no external migration runner is required.

For schema changes on an existing database (e.g., adding a column), SQLite's ALTER TABLE … ADD COLUMN is used, wrapped in try/catch so re-running on an already-migrated database silently ignores the error.

15. Design Decisions

DecisionRationale
SQLite instead of MySQL/PostgreSQLZero-config embedded DB suitable for Phase 1. No DB server needed. Easy to set up for demo and evaluation. Can be migrated to PostgreSQL in Phase 2 with minimal code change.
node:sqlite DatabaseSync (built-in)No additional npm dependency. Synchronous API simplifies controller code — no async/await complexity for DB calls.
Separate roles table (FK from users)Normalised design. Adding or renaming a role requires one row change, not a CHECK constraint migration. Also avoids the sparse-column problem if role-specific fields are ever needed.
Separate categories table (FK from complaints)Categories can be added or renamed by admin without a frontend deployment. Enforces referential integrity — a complaint cannot reference a non-existent category.
complaint_id as TEXT (CMP-001 format)Human-readable IDs are more useful in a support context than auto-incremented integers for referencing in communication.
assigned_to as INTEGER FK to users.idProper referential integrity — the system cannot assign a complaint to a user who doesn't exist. Joined at query time to get agent name.
SLA deadline stored when priority is set (not at creation)Priority is not known at submission — it is assessed by admin/supervisor after reviewing the complaint. Starting the SLA clock at submission without a priority would be meaningless.
Closing rules based on feedback ratingClosing rights depend on whether the customer was satisfied. A dissatisfied customer (rating 1–2) requires management review before closure; a satisfied one (3–5) allows agent to close. No feedback within 7 days triggers admin/supervisor force-close.
Notifications as DB table (not WebSocket)Simpler to implement. 30s polling is acceptable for this use case. Phase 2 can add WebSocket real-time push.
CHECK constraints on status and priorityData integrity enforced at the DB layer, not just application layer. Prevents invalid data even if API is bypassed.
agent_categories as junction table (not a text column on users)Normalised many-to-many design. An agent can cover multiple categories; a category can have multiple agents. A text column on users could not express this. The junction table also enables direct queries — finding all agents for a category or all categories for an agent is a single indexed lookup.
cross_category_assignments as separate table (not a boolean on complaint_history)A boolean is_cross_category column on complaint_history would be sparse — almost all rows would be 0. A separate table stores only meaningful data, enables direct COUNT queries without text parsing, and captures full detail (bypassed category, assigner, reason) that would be awkward to fit into the existing history schema. The note NOT NULL DB constraint guarantees no override can ever be recorded without a reason, even if application validation is bypassed.
